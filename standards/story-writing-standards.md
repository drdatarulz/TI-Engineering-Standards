# Story Writing Standards

## Purpose

This standard defines how user stories and development tickets are written, structured, and ordered for TI projects. It applies to both bulk PRD decomposition and incremental story additions. All stories — whether generated by Claude Code or written by hand — follow these conventions.

This standard covers only the *shape and structure* of stories. For board mechanics, see `project-tracking.md`. For implementation, see `architecture.md` and the [implement-ticket-v3 skill](../skills/implement-ticket-v3/SKILL.md). For testing, see `testing.md` and the [integration-test-v3 skill](../skills/integration-test-v3/SKILL.md).

## Core Principle: Vertical Slices Over Horizontal Layers

Stories should deliver user-visible functionality whenever possible. A vertical slice cuts through all layers of the stack (database, domain, infrastructure, API, UI) for a single capability. After completing a vertical slice, there is something a human can see, interact with, or verify.

**Prefer this:**
- "User can log in and see the dashboard with summary counts"
- "User can create a new record, see it in the list, view its detail, and delete it"
- "User can filter the record list by status and date range"

**Avoid this:**
- "Create database tables for all entities"
- "Build all domain models and interfaces"
- "Add all API endpoints"
- "Build the UI"

Horizontal stories (pure infrastructure, pure database, pure plumbing) are acceptable only when they are genuine prerequisites that cannot be folded into a vertical slice. When horizontal stories are necessary, they should be bounded to the minimum needed to unblock the first vertical slice.

## Story Sizing

Stories should be completable by a single developer agent session. The guiding principle is **group by entity lifecycle, not by individual operation.** The basic CRUD operations for an entity (create, view/detail, list, delete) should travel together in one story because they are only useful together — you can't meaningfully demonstrate "create" without being able to see what you created.

### Sizing Examples

- **Too small:** "Add the Name column to the Records table" or "Add a create record endpoint" or "Add a delete button to the record list." These are fragments. A story per CRUD operation produces tickets that individually deliver nothing demonstrable. Fold them into the entity lifecycle story.
- **Right size:** "User can create a new record, see it in the record list, view its detail, and delete it." This is the base entity lifecycle — after this one story, you can exercise the full basic flow. It touches database, domain, API, and UI for one cohesive capability.
- **Also right size:** "User can edit a record's name, category, and description." Edit is often a separate story when the edit form has meaningfully different complexity (validation, multi-step, conditional fields) from the create form. If edit is simple and mirrors create, fold it into the lifecycle story.
- **Too large:** "User can manage all aspects of records including create, edit, delete, bulk operations, import, export, and audit history." Break this apart — but along capability boundaries, not individual operations.

### How to Split a Story That's Too Large

Split by **capability layer**, not by CRUD operation:

1. Base entity lifecycle: create, list, view detail, delete (one story)
2. Edit capability (one story, if sufficiently complex to warrant separation)
3. Filtering and search (one story)
4. Export (one story)
5. Bulk operations (one story)
6. Audit history (one story)

Each story in this progression builds on the previous one and is independently demonstrable. After each, there is something new a human can see and verify.

## Milestones

Milestones are review gates — points in the backlog where the application is runnable and a human reviews what's been built. They serve the same purpose as a sprint review: prevent work from accumulating too long without validation.

### Milestone Rules

- Every backlog has at least one milestone (even if it's just "build everything, then review")
- Milestone 1 should be reachable quickly — aim for the minimum set of stories that produces something visible and interactive (e.g., "login works, dashboard renders")
- Each milestone ends with a **milestone marker issue** that the orchestrator recognizes as a hard stop
- Stories within a milestone can be processed autonomously; the milestone boundary is the human checkpoint
- Milestones map to screen inventory groupings when a screen inventory exists

### Milestone Marker Issue Format

Milestone markers use the `milestone` label and follow this structure in the issue body:

```markdown
## Review Checklist
- [ ] Application starts without errors
- [ ] [Screen/flow 1] is functional: [brief description of what to verify]
- [ ] [Screen/flow 2] is functional: [brief description of what to verify]
- [ ] No regressions in previously completed milestones

## Smoke Test
- [ ] `dotnet run` starts the application successfully
- [ ] GET /health returns 200
- [ ] [Key endpoint] returns expected response
- [ ] UI loads at localhost:[port] without console errors
```

The orchestrator treats issues with the `milestone` label as hard stops regardless of operating mode. It runs the smoke test, reports results, and waits for human approval before continuing.

### Linking Stories to Milestones via Sub-Issues

After creating both the milestone marker and its stories, **link each story as a sub-issue of the milestone marker** using GitHub's sub-issues feature. This gives the milestone epic-style progress tracking — GitHub displays "N of M complete" on the parent issue as child stories are closed.

To link a story as a sub-issue:

1. Get the story's database ID (not the issue number):
   ```bash
   gh api graphql -f query='query($owner: String!, $repo: String!, $number: Int!) {
     repository(owner: $owner, name: $repo) { issue(number: $number) { databaseId } }
   }' -f owner={OWNER} -f repo={REPO} -F number={STORY_ISSUE_NUMBER} --jq '.data.repository.issue.databaseId'
   ```
2. Add it as a sub-issue of the milestone marker using the GitHub MCP `sub_issue_write` tool (method: `add`, issue_number: milestone issue number, sub_issue_id: story database ID).

This applies to both bulk PRD decomposition and incremental story additions. Every story that belongs to a milestone must be linked as a sub-issue of that milestone's marker.

## Backlog Ordering

### Phase 1: Foundation (Horizontal — Keep Minimal)

Limited to work that genuinely cannot be part of a vertical slice:

- Project scaffolding (solution structure, CI configuration)
- Shared infrastructure (auth middleware, logging pipeline, error handling middleware)
- Database project setup and initial seed migration
- Docker test environment configuration

Foundation should be as small as possible. If a database table is only used by one feature, it belongs in that feature's vertical slice, not in a foundation "create all tables" ticket.

Use the `foundation` label for these stories.

### Phase 2+: Vertical Slices Grouped by Milestone

After foundation, all stories should be vertical slices delivering user-visible functionality. Group them by milestone, with the most foundational user flows first:

- **Milestone 1:** Authentication + primary landing screen. The goal is "I can log in and see something."
- **Milestone 2:** Core entity CRUD — the primary thing the application manages. "I can create, view, edit, and delete the main thing."
- **Milestone 3+:** Secondary features, reporting, integrations, polish.

Within a milestone, order stories so that each builds on the previous one logically. If story B depends on story A's screen or data, A comes first.

## Acceptance Criteria

Acceptance criteria are the contract between the story and the developer. They are also the verification targets for the integration tester and the engineering reviewer.

### Criteria Rules

**Specific and testable.** Each criterion should be verifiable by either a human looking at the running application or an automated test. "User experience is good" is not testable. "Dashboard loads in under 2 seconds with 1000 records" is testable.

**Behavior-focused, not implementation-focused.** Describe what happens, not how it's built. "When the user submits an empty form, a validation message appears below the name field" — not "add FluentValidation to the CreateRecordRequest class."

**Complete.** Cover the happy path, key error cases, and edge cases. If a form has validation, the criteria should specify what happens when validation fails, not just what happens when it succeeds.

**Checkboxable.** Each criterion is a checkbox (`- [ ]`) that can be marked done independently. The orchestrator checks these off when a story completes.

## Incremental Story Additions

When adding stories to an existing backlog (not during initial PRD decomposition):

1. Review the existing backlog and codebase to understand current patterns and where the project stands
2. Write the story following the same structure and conventions above
3. Determine placement: does it attach to an existing milestone, or does it need a new one?
4. Check for dependencies on existing stories (both directions — does anything depend on this?)
5. If the new story introduces a new screen or flow, update the screen inventory
6. Default to treating incremental additions as their own review gate unless they're clearly minor

## Labels

| Label | Meaning |
|-------|---------|
| `story` | User-visible functionality (vertical slice) |
| `task` | Operational or non-code work |
| `bug` | Defect in existing functionality |
| `foundation` | Infrastructure/scaffolding (horizontal, pre-milestone-1 only) |
| `milestone` | Milestone marker issue (review gate) |
